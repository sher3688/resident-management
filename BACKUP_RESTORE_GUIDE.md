# 社區管理系統 — 資料備份與回復指南

## 概述

社區管理系統提供完整的資料備份與回復功能，確保住戶資料與報修記錄的安全性。所有備份與回復操作均需管理者權限。

---

## 1. 資料備份

### 1.1 自動備份

系統每次資料變更（新增、編輯、刪除）時，均自動保存至雲端資料庫。無需手動操作。

### 1.2 手動備份

**步驟：**

1. 登入系統，確認您具有**管理者**權限
2. 點擊左側邊欄下方的 **⚙️ 管理者設定**
3. 選擇 **資料備份** 標籤
4. 點擊 **下載備份** 按鈕
5. 系統將自動下載 JSON 格式的備份檔案（檔名：`backup-YYYY-MM-DD.json`）

**備份檔案內容：**

```json
{
  "timestamp": "2026-06-20T12:34:56.789Z",
  "residents": [
    {
      "id": 1,
      "unitNumber": "A01",
      "ownerName": "王小明",
      "ownerPhone": "0912-345-678",
      ...
    }
  ],
  "repairRequests": [
    {
      "id": 1,
      "unitNumber": "A01",
      "repairDate": "2026-06-20",
      ...
    }
  ],
  "totalRecords": 551
}
```

### 1.3 備份建議

- **頻率：** 每週至少備份一次，重要變更後立即備份
- **存儲位置：** 將備份檔案存放在安全的位置（如 Google Drive、Dropbox、NAS）
- **版本管理：** 保留至少 3 個月的備份檔案，便於追蹤歷史變更

## 2. 資料回復

### 2.1 回復住戶資料

**使用場景：**
- 誤刪住戶資料
- 需要恢復舊版本住戶資訊
- 系統遷移或轉移

**步驟：**

1. 登入系統，確認您具有**管理者**權限
2. 點擊左側邊欄下方的 **⚙️ 管理者設定**
3. 選擇 **資料備份** 標籤
4. 在 **恢復資料** 區塊，點擊 **選擇檔案** 並選取備份 JSON 檔案
5. 點擊 **恢復住戶資料** 按鈕
6. 系統將驗證檔案格式並逐筆匯入
7. 完成後顯示成功/失敗統計

**注意事項：**
- 回復操作**不會刪除**現有資料，而是**新增或更新**記錄
- 若住戶戶號已存在，系統將更新該戶的資訊
- 若住戶戶號不存在，系統將新增該戶

### 2.2 回復報修記錄

**使用場景：**
- 誤刪報修記錄
- 需要恢復舊版本報修資訊

**步驟：**

1. 登入系統，確認您具有**管理者**權限
2. 點擊左側邊欄下方的 **⚙️ 管理者設定**
3. 選擇 **資料備份** 標籤
4. 在 **恢復資料** 區塊，點擊 **選擇檔案** 並選取備份 JSON 檔案
5. 點擊 **恢復報修記錄** 按鈕
6. 系統將驗證檔案格式並逐筆匯入
7. 完成後顯示成功/失敗統計

## 3. 常見問題

### Q: 備份檔案損壞或無法讀取怎麼辦？

**A:** 
- 確認檔案格式為有效的 JSON（可用線上 JSON 驗證工具檢查）
- 嘗試使用較舊的備份檔案
- 若所有備份均損壞，請聯繫系統管理員

### Q: 回復時出現「失敗 X 筆」的錯誤訊息

**A:**
- 系統會自動跳過無效或重複的記錄
- 檢查錯誤詳情（通常顯示前 3 筆失敗記錄）
- 常見原因：戶號格式不符、必填欄位缺失、資料型別錯誤
- 可手動修正備份檔案後重新嘗試

### Q: 能否恢復特定時間點的資料？

**A:**
- 系統不提供時間點恢復功能
- 建議定期備份並標記備份時間，需要時手動選擇對應的備份檔案
- 若需要精細的版本控制，可考慮使用外部版本管理工具

### Q: 備份檔案應該多久備份一次？

**A:**
- **最低建議：** 每週一次
- **高風險環境：** 每天一次或每次重要變更後立即備份
- **長期保存：** 保留至少 3 個月的備份，重要檔案保留 1 年以上

## 4. 安全建議

### 4.1 備份檔案安全

- **加密存儲：** 將備份檔案存放在加密的雲端服務（如 Google Drive、OneDrive）
- **多地備份：** 在至少 2 個不同的位置存放備份
- **定期驗證：** 每月驗證一次備份檔案的完整性（嘗試恢復到測試環境）
- **存取控制：** 限制備份檔案的存取權限，僅授予必要人員

### 4.2 操作安全

- **權限控制：** 僅授予信任的管理者進行備份與回復操作
- **操作紀錄：** 系統自動記錄所有備份與回復操作（包括時間、操作者、結果）
- **測試環境：** 在生產環境前，先在測試環境驗證回復流程

### 4.3 災難恢復計畫

建議制定完整的災難恢復計畫，包括：

1. **備份策略：** 定期備份頻率、保留期限、存儲位置
2. **恢復流程：** 清晰的恢復步驟、責任人、時間目標
3. **測試計畫：** 定期測試備份與恢復功能（至少每季一次）
4. **聯絡清單：** 緊急聯絡人、技術支援、系統管理員

## 5. 技術細節

### 5.1 備份檔案格式

備份檔案採用 JSON 格式，包含以下頂層欄位：

| 欄位 | 型別 | 說明 |
|---|---|---|
| `timestamp` | string (ISO 8601) | 備份建立時間 |
| `residents` | array | 所有住戶資料 |
| `repairRequests` | array | 所有報修記錄 |
| `totalRecords` | number | 總記錄數（residents + repairRequests） |

### 5.2 住戶資料欄位

| 欄位 | 型別 | 說明 |
|---|---|---|
| `id` | number | 住戶唯一識別碼 |
| `unitNumber` | string | 戶號 |
| `ownerName` | string | 區權人姓名 |
| `ownerPhone` | string | 區權人電話 |
| `coResident1Name` | string | 同住人 1 姓名 |
| `coResident1Phone` | string | 同住人 1 電話 |
| `carParkingNumber` | string | 汽車車位號碼 |
| `carPlateNumber` | string | 汽車牌照號碼 |
| `motorcycleParkingNumber` | string | 機車車位號碼 |
| `motorcyclePlateNumber` | string | 機車牌照號碼 |
| `address` | string | 住戶住所 |
| `emergencyContactName` | string | 緊急聯絡人 1 姓名 |
| `emergencyContactPhone` | string | 緊急聯絡人 1 電話 |
| `emergencyContactAddress` | string | 緊急聯絡人 1 地址 |
| `squareMeters` | string | 坪數 |
| `waterMeterNumber` | string | 水號 |
| `electricityMeterNumber` | string | 電號 |
| `moveInDate` | string (YYYY-MM-DD) | 入住日期 |
| `createdAt` | string (ISO 8601) | 建立時間 |
| `updatedAt` | string (ISO 8601) | 最後修改時間 |

### 5.3 報修記錄欄位

| 欄位 | 型別 | 說明 |
|---|---|---|
| `id` | number | 報修唯一識別碼 |
| `unitNumber` | string | 報修戶號 |
| `repairDate` | string (YYYY-MM-DD) | 報修日期 |
| `reporterName` | string | 報修人姓名 |
| `reporterPhone` | string | 報修人電話 |
| `location` | string | 報修位置 |
| `description` | string | 狀況描述 |
| `status` | string | 修繕進度（待處理/處理中/已完成/已取消） |
| `notes` | string | 處理備註 |
| `createdAt` | string (ISO 8601) | 建立時間 |
| `updatedAt` | string (ISO 8601) | 最後修改時間 |
